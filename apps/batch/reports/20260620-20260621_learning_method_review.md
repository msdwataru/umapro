# 学習方法見直し 検証レポート

**期間**: 2026-06-20 〜 2026-06-21  
**目的**: Phase1 評価で判定「B: 優位性なし」を受け、学習方法の根本的見直しを行う  
**最終データ**: `artifacts/race_features_v1.parquet`（2022-01-05 〜 2026-06-20、修正済）  
**評価期間**: 2026-01-15 以降（Walk-Forward Fold5 out-of-sample）  

> **注意**: ROI はすべて OPTIMISTIC_BIAS（確定オッズ近似値による楽観値）。  
> `odds_at_bet`（T-5min）が未収集のため、真の ROI はさらに低い可能性がある。

## 1. Phase1 評価結果（前提）

| モデル | 評価期間 | 購入数 | ECE | ROI | Phase1 判定 |
|---|---|---|---|---|---|
| course_form（ルールベース） | 2020-2026 全期間 | 10,244 | 0.0834 | -19.6% | B: 優位性なし |
| lgbm_ranker v1（LightGBM Ranker） | 2026年アウトサンプル | 1,572 | 0.2285 | -22.8% | B: 優位性なし |

**根本原因**: `rank_xendcg` はランキング順位を最適化するが確率推定をしない → ECE=0.229（5%と言っているが実勝率 30%）、EV-ROI 相関=-0.04（機能しない）

## 2. オッズ除外 LightGBM Ranker（2026-06-20 実施）

**仮説検証**: モデルがオッズから勝率を逆算しているだけか？  
`latest_win_odds`, `odds_inv`, `odds_rank` を完全除外して Ranker を学習（旧データ 82K 行）。

| 指標 | 値 |
|---|---|
| best_iteration | 21（500 で早期収束） |
| ECE | **0.0036**（大幅改善） |
| ROI | **-27.4%** / 95% CI [-35.2%, -19.1%] |
| p値（vs Market） | 1.0000 |
| Market ROI | -20.2% |

**結論**: ECE は劇的に改善したが ROI は悪化。「キャリブレーション改善 ≠ ROI 改善」を確認。  
best_iter=21 の早期収束は「信号不足」または「データ量不足」のどちらかで要切り分け。

---

## 4. 専門家レビューによる方針転換（2026-06-21 受領）

> 「Overlay 直接学習は単独では機能しない。今の特徴量セットは `odds_inv` が支配的 = 入力が市場確率そのもの。  
>  市場確率を入力にして『実勝率 - 市場確率』を予測させると、残差 0 が最適解になる。  
>  **Overlay 学習が機能する必須条件は、入力からオッズ系を除外または直交化することが先。**」

正しい打ち手の順序：

| Step | 施策 |
|---|---|
| Step 1 | データ年代・信号量の切り分け |
| Step 2 | 欠損データ修正 → 特徴量再生成 |
| Step 3 | オッズ除外 Binary で再学習（データ量仮説の検証） |
| Step 4 | 市場直交性が高い特徴量の追加 |
| Step 5 | Overlay 直接ターゲット学習（Step 3 結果を踏まえて設計） |

## 5. インフラ修正（2026-06-21 午後）

### 5-1. 特徴量再生成タイムアウト問題と修正

`uma/features/builder.py` の `_load_target_entries` が全 race_entries を取得してから Python 側で日付フィルタしていたため、Supabase のステートメントタイムアウト（57014）が発生。

**修正内容 (`uma/db/client.py`)**:

| 変更 | 内容 |
|---|---|
| `paginate()` | タイムアウト・接続断に対してリトライ＋指数バックオフ追加（最大 5回） |
| `paginate_keyset()` | 新規追加。`WHERE id > last_id ORDER BY id` 方式でオフセット O(n) 問題を解消 |

**修正内容 (`uma/features/builder.py`)**:

| 関数 | 変更 |
|---|---|
| `_load_target_entries` | ① races テーブルから対象 race_id を取得 → ② IN 句で絞り込み（全件取得廃止） |
| `_load_all_results` | offset ページネーション → キーセットページネーション（race_entry_id 順）に変更 |

## 7. パターン C の収束確認（n=1000）

C が 500 で上限到達・未収束のため n=1000 で追加実行。

| 指標 | n=500 | **n=1000** |
|---|---|---|
| best_iteration | 500（上限） | **998（上限 ≒ 未収束）** |
| LogLoss | 0.2133 | **0.2106**（改善継続） |
| ROI | -20.4% | **-21.5%** |
| 95% CI | [-26.9%, -13.8%] | [-28.2%, -14.9%] |
| p値 | 1.0000 | 1.0000 |
| vs Market | +0.15pt | **-0.94pt** |

**判定**: LogLoss は改善しているのに ROI が悪化 → 「キャリブレーション改善が ROI に直結しない」の繰り返し確認。  
**log(odds) 重み付けの天井は Market 前後（-20〜-21%）であり、アプローチを変えないと Market を有意に超えることは不可能と結論。**

## 9. 総合結論と次のステップ

### 結論

| 検証項目 | 結論 |
|---|---|
| 目的関数変更（Ranker → Binary） | ECE は改善、ROI は不変 |
| オッズ除外 | データ量増加で best_iter は伸びるが ROI は改善せず |
| log(odds) 重み付け | Market と同等水準まで改善、ただし有意差なし。天井は Market 前後 |
| データ量不足仮説 | 部分的に正（best_iter が 21 → 219 に改善）、ただし ROI 改善には不十分 |
| 信号不足仮説 | 残存する（非市場特徴量の win 相関は市場の半分以下） |

### 次のアクション：Overlay 直接ターゲット学習

専門家処方の Step 3 へ進む。

```
target = actual_win(0/1) - market_prob(1/odds 正規化)
features = odds 除外の非市場特徴量のみ
```

- 市場の誤りを直接回帰させることで、モデルが「市場が過小評価している馬」を学習
- Overlay ≥ 15% で正の ROI の兆候（B モデル）があり、方向性の根拠あり
- `uma/models/train_overlay.py` として実装予定
